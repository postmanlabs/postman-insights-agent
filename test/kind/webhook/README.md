## Reproduce the 5c.3b verification

```sh
# 1. Build the agent image (bundles Java agent JAR at /opt/postman-java-agent.jar)
cd <repo-root>
docker build -f test/kind/Dockerfile.agent -t postman-insights-agent:5c3b .
kind load docker-image postman-insights-agent:5c3b --name pia-https-test

# 2. Generate dev TLS + render manifests
cd test/kind/webhook
./gen-tls-and-manifests.sh

# 3. Apply (rollback command: kubectl delete mutatingwebhookconfiguration postman-insights-webhook)
kubectl apply -f webhook-deployment.yaml
kubectl wait --for=condition=available --timeout=60s \
    deploy/postman-insights-webhook -n postman-insights
kubectl apply -f webhook-config.yaml

# Java capture: use the main DaemonSet (not a separate javatls-capture Deployment)
kubectl apply -f ../agent-daemonset.yaml
kubectl rollout restart -n postman-insights daemonset/postman-insights-agent

# 4. Create opted-in namespace + a Java test pod
kubectl create ns test-webhook
kubectl label ns test-webhook postman.dev/insights=enabled
kubectl run java-test --image=tomcat:10-jdk21 -n test-webhook
kubectl get pod java-test -n test-webhook \
    -o jsonpath='{.spec.containers[0].env}'
# → should contain JAVA_TOOL_OPTIONS=-javaagent:/postman/postman-java-agent.jar
```

Full evidence in `docs/phases/phase-5c3b-results.md`.

## Teardown

```sh
# THE one command that undoes ALL webhook involvement:
kubectl delete mutatingwebhookconfiguration postman-insights-webhook

# Full cleanup
kubectl delete -n postman-insights deploy/postman-insights-webhook \
    svc/postman-insights-webhook secret/postman-insights-webhook-tls
kubectl delete -n postman-insights deploy/javatls-capture --ignore-not-found
kubectl delete ns test-webhook
```

## Production caveat

**These certs are dev-only.** Phase 5c.3c converts this to a Helm chart
with cert-manager bootstrap. Don't use this directory's TLS material in
any cluster you care about.
