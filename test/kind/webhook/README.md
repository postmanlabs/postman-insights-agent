# test/kind/webhook — local kind testing for the mutating admission webhook

Kind-cluster end-to-end artifacts for the Postman Insights mutating admission
webhook. This directory covers **local dev testing only**. For production
deployments use the Helm chart in
[`charts/postman-insights-webhook/`](../../../charts/postman-insights-webhook/).

## Files committed to the repo

| File | Purpose |
| --- | --- |
| `gen-certs.sh`                   | Generate self-signed TLS certs and render `webhook-deployment-with-certs.yaml` |
| `gen-tls-and-manifests.sh`       | Older cert-gen script (kept for reference; `gen-certs.sh` supersedes it) |
| `openssl.cnf`                    | OpenSSL config for the webhook server CSR (SAN entries for in-cluster DNS) |
| `webhook-deployment.yaml.tmpl`   | Secret + Deployment + Service template (cert/key inlined at render) |
| `webhook-config.yaml.tmpl`       | `MutatingWebhookConfiguration` template (CA bundle inlined at render) |
| `capture-deployment.yaml`        | **Deprecated** — Java capture is on the `postman-insights-agent` DaemonSet (`--enable-java-tls`) |
| `README.md`                      | This file |

## Files generated locally (gitignored — never commit)

| File | Why gitignored |
| --- | --- |
| `ca.key`, `ca.crt`, `ca.srl`                    | Dev CA private key + paired cert |
| `tls.key`, `tls.crt`                            | Webhook server key + cert (output by `gen-certs.sh`) |
| `webhook.key`, `webhook.csr`, `webhook.crt`     | Webhook server key + cert (output by `gen-tls-and-manifests.sh`) |
| `webhook-deployment-with-certs.yaml`            | Rendered deployment with inlined certs |
| `webhook-deployment.yaml`, `webhook-config.yaml` | Rendered manifests from the older `.tmpl` workflow |

All of the above are regenerated by `gen-certs.sh`.

## Quick start

```sh
# 1. Build the agent image (bundles the Java agent JAR at /opt/postman-java-agent.jar)
cd <repo-root>
docker build -f build-scripts/Dockerfile.ebpf -t postman-insights-agent:local .
kind load docker-image postman-insights-agent:local --name <your-kind-cluster>

# 2. Generate dev TLS + render the manifest
cd test/kind/webhook
./gen-certs.sh

# 3. Apply
kubectl apply -f webhook-deployment-with-certs.yaml
kubectl wait --for=condition=available --timeout=60s \
    deploy/postman-insights-webhook -n postman-insights

# Apply the MutatingWebhookConfiguration (uses gen-certs.sh CA bundle)
# Note: gen-certs.sh embeds the CA into the rendered manifest above; if using
# the .tmpl workflow instead, apply webhook-config.yaml here.

# Java capture: use the main DaemonSet, not a separate deployment
kubectl apply -f ../agent-daemonset.yaml
kubectl rollout restart -n postman-insights daemonset/postman-insights-agent

# 4. Create an opted-in namespace + a Java test pod
kubectl create ns test-webhook
kubectl label ns test-webhook postman.com/insights=enabled
kubectl run java-test --image=tomcat:10-jdk21 -n test-webhook
kubectl get pod java-test -n test-webhook \
    -o jsonpath='{.spec.containers[0].env}'
# → should contain JAVA_TOOL_OPTIONS=-javaagent:/postman/postman-java-agent.jar
```

## Teardown

```sh
# The one command that undoes all webhook involvement:
kubectl delete mutatingwebhookconfiguration postman-insights-webhook

# Full cleanup
kubectl delete -n postman-insights deploy/postman-insights-webhook \
    svc/postman-insights-webhook secret/postman-insights-webhook-tls
kubectl delete ns test-webhook
```

## Production caveat

**These certs are dev-only.** For production, use the Helm chart in
[`charts/postman-insights-webhook/`](../../../charts/postman-insights-webhook/)
which supports cert-manager or a pre-existing TLS Secret.
