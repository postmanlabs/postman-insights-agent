# CLAUDE.md

Guidance for Claude Code (and other AI coding agents) when working in this repository.

## Project overview

This is the **Postman Insights Agent** ‚Äî an open-source Go CLI that captures HTTP traffic from network interfaces (via libpcap), parses/witnesses requests and responses, and ships them to the Postman Insights backend so users can see endpoints, errors, latency, and volume in Postman.

- **Language:** Go (see `go.mod` ‚Äî `go 1.25.0`, toolchain `go1.25.11`).
- **Module:** `github.com/postmanlabs/postman-insights-agent`
- **Entry point:** `main.go` ‚Üí `cmd.Execute()` (Cobra-based CLI).
- **Binary name:** `postman-insights-agent`
- **License:** See `LICENSE`.
- **Origin:** Forked from / successor to the Akita Software agent ‚Äî many internal packages still reference `akita*` and `akitasoftware/*` libs.

Note: This open-source repo does **not** include type/format inference. That functionality only exists in the official distributed binary.

## Build, test, run

System prerequisites:
- Go 1.25 (repo pins `go1.25.0` / toolchain `go1.25.11`).
- `libpcap`
  - macOS: `brew install libpcap`
  - Debian/Ubuntu: `apt-get install libpcap-dev`
- `mockgen` for tests: `go install github.com/golang/mock/mockgen@v1.5.0`
- For eBPF features: `clang`, `llvm`, `linux-headers` (CI installs these automatically; see `.circleci/config.yml`).

Common commands (from `Makefile`):

| Command         | What it does |
| --------------- | ------------ |
| `make`          | `make build` ‚Äî produces `bin/postman-insights-agent` |
| `make build`    | `go build -o bin/postman-insights-agent .` (runs `clean` first) |
| `make clean`    | `go clean` |
| `make mock`     | `go generate ./rest` (regenerates gomock mocks) |
| `make generate-ebpf` | Generate bpf2go bindings (`libssl_*_bpfel.go`) inside the Linux dev container; use on macOS before eBPF IDE work or to fix gopls `undefined: libsslObjects` errors (requires `make dev-build` + Docker) |
| `make test`     | `make mock` then `go test ./...` |
| `make docker-build` | Build via `build-scripts/Dockerfile`, output binary to `bin/` |

CI (`.circleci/config.yml`) runs `make` and `gotestsum --junitfile ...` against all packages.

## Repository layout

Top-level CLI plumbing:
- `main.go` ‚Äî trivial entry point.
- `cmd/` ‚Äî Cobra command tree (`root.go`, `supervisor.go`).
- `cmd/internal/` ‚Äî subcommand implementations:
  - `apidump/` ‚Äî `apidump` command: the core traffic-capture command.
  - `apidump-ebpf/` ‚Äî hidden `apidump-ebpf` command: credential-free eBPF capture for local dev/debugging.
  - `ec2/` ‚Äî `ec2 setup|remove`: install agent as a systemd service on EC2.
  - `ecs/` ‚Äî `ecs add|remove|cf-fragment|task-def`: AWS ECS integration.
  - `kube/` ‚Äî `kube inject|run|secret|helm-fragment|tf-fragment|webhook`: Kubernetes integration (DaemonSet, sidecar injection, manifests, mutating webhook).
  - `kube-webhook/` ‚Äî `kube webhook` subcommand: HTTPS admission webhook server that auto-injects the Java agent into pods.
  - `legacy/` ‚Äî legacy `specs` and other deprecated commands.
  - `ascii/`, `akiflag/`, `cmderr/`, `pluginloader/` ‚Äî CLI utilities.

Core domain packages:
- `apidump/` ‚Äî orchestrates a capture session: pcap ‚Üí parse ‚Üí trace ‚Üí backend.
- `pcap/` ‚Äî libpcap wrappers, packet/stream reassembly, replay.
- `learn/` ‚Äî HTTP parsing (request/response ‚Üí IR "witnesses"), JSON preprocessing, event-stream parsing, Luhn checks.
- `trace/` ‚Äî collectors (`backend_collector`, `dummy_collector`), rate limiting, stats, filters, reporting buffer.
- `rest/` ‚Äî HTTP clients for the Postman/Akita backend (`front_client`, `learn_client`, base client, auth, errors). Mocks are generated here via `go generate`.
- `daemon/` ‚Äî long-running daemon HTTP server.
- `plugin/` ‚Äî pluggable architecture (`interface.go`, `akita/` plugin).
- `ebpf/` ‚Äî eBPF-based HTTPS capture (libssl uprobes). Key sub-packages:
  - `ebpf/loader/` ‚Äî loads BPF programs via `bpf2go`; generated `*_bpfel.go` files are gitignored (regenerated by CI).
  - `ebpf/events/` ‚Äî Go-side event types, ring-buffer reader, HTTP/2 adapter.
  - `ebpf/programs/` ‚Äî BPF C source (`libssl.bpf.c`, `event.h`).
  - `ebpf/discovery/` ‚Äî /proc-based PID watcher for auto-discovery of libssl processes.

Integrations and platform helpers:
- `integrations/cri_apis/` ‚Äî Container Runtime Interface client (containerd/CRI-O).
- `integrations/kube_apis/` ‚Äî Kubernetes API access.
- `integrations/nginx/` ‚Äî nginx-related helpers.
- `integrations/tests/` ‚Äî integration tests.
- `aws_utils/` ‚Äî AWS SDK helpers.
- `tcp_conn_tracker/`, `tls_conn_tracker/` ‚Äî connection tracking.
- `useragent/`, `location/`, `version/`, `setversion/`, `telemetry/`, `usage/`, `printer/`, `consts/`, `cfg/`, `env/`, `util/` ‚Äî utilities.

Other:
- `data_masks/` ‚Äî PII/sensitive data redaction.
- `apispec/` ‚Äî API spec generation/handling.
- `architecture/architecture.go` ‚Äî architecture metadata.
- `docs/discovery-mode.md` ‚Äî important user-facing doc explaining **Discovery Mode** vs **Workspace Mode** onboarding (Kubernetes). Read this before changing onboarding/k8s flows.
- `charts/postman-insights-webhook/` ‚Äî Helm chart for the mutating admission webhook (Java agent auto-injection). This is the canonical chart location; do not put charts under `deployment/`.
- `deployment/` ‚Äî Go package (`package deployment`) for deployment-environment detection (EC2, ECS, k8s, etc.). **Not** for chart artifacts.
- `build-scripts/`, `ci/` ‚Äî release & deployment scripts. `build-scripts/Dockerfile` uses `golang:1.25-alpine` for static binary builds (Alpine's libpcap has no D-Bus dependency).

## CLI surface (high level)

Root: `postman-insights-agent` (see `cmd/root.go`). Notable subcommands:

- `apidump` ‚Äî Capture requests/responses from network traffic (primary command). Supports `--enable-https-capture` (eBPF libssl).
- `ec2 setup` / `ec2 remove` ‚Äî Install/uninstall as systemd service on EC2.
- `ecs add|remove|cf-fragment|task-def` ‚Äî Manage agent on AWS ECS.
- `kube inject|run|secret|helm-fragment|tf-fragment` ‚Äî Kubernetes integration. `kube run` is the DaemonSet entrypoint (Linux only).
- `kube webhook` ‚Äî Run the mutating admission webhook that auto-injects the Postman Java agent into pods in opted-in namespaces. Runs as a separate Deployment in the cluster (not the DaemonSet); deployed via `charts/postman-insights-webhook/`.
- `specs` (legacy) ‚Äî Manage API specs.
- `aki` ‚Äî ASCII art easter egg.

Test-only / hidden flags live on the root command (`testOnlyUseHTTPSFlag`, `dogfoodFlag`, `debugFlag`, profiling flags, etc.) ‚Äî see `cmd/root.go`.

## Conventions and gotchas

- **Cobra + Viper + pflag** are used throughout. New subcommands should follow the pattern in `cmd/internal/<name>/<name>.go` and be wired into `cmd/root.go`.
- **Mocks**: `rest/` uses gomock. Run `make mock` (or `go generate ./rest`) after changing interfaces in `rest/interface.go`. `make test` does this automatically.
- **libpcap is required at build time** on every platform ‚Äî `pcap/get_pcap_handle_linux.go` is Linux-only; the generic file covers other OSes.
- **Linux-only features**: `kube run` (DaemonSet mode), eBPF capture, and some pcap paths assume Linux. Be mindful when adding code under build-tagged files.
- **eBPF generated files**: `ebpf/loader/*_bpfel.go` and `*.o` are produced by `go generate ./ebpf/loader` (via `bpf2go`) and are gitignored ‚Äî CI regenerates them from source. Do not commit them. On macOS, run `make dev-build` then `make generate-ebpf` to create them locally for VS Code/gopls (see `ebpf/README.md`).
- **Helm charts** live in `charts/`, not `deployment/`. The `deployment/` directory is a Go package for environment detection.
- **Akita lineage**: Many package names, struct names, and dependencies still say `akita`. Do **not** rename these casually ‚Äî they cross module boundaries (`github.com/akitasoftware/akita-ir`, `akita-libs`, `go-utils`). When in doubt, leave the name alone.
- **Backend talk** goes through `rest/` clients. Avoid hitting external HTTP from other packages directly.
- **Telemetry** is in `telemetry/`; usage events in `usage/`. Keep new event names consistent with existing ones.
- **Discovery vs Workspace mode**: see `docs/discovery-mode.md`. New k8s onboarding work should support both and avoid resurrecting the legacy `--project` / `--collection` flags.
- **Code owners**: see `.github/CODEOWNERS` before opening PRs.
- **Versioning**: SemVer; bump via files under `version/` and `setversion/` as documented in `CONTRIBUTING.md`.

## Running locally

```bash
make                                  # builds bin/postman-insights-agent
./bin/postman-insights-agent --help   # explore commands
./bin/postman-insights-agent apidump --help
```

Capturing traffic typically requires root / `CAP_NET_RAW` (libpcap). Use `sudo` locally or run inside the provided container.

## When making changes

1. Prefer small, well-scoped edits. The codebase mixes legacy Akita code with newer Postman code ‚Äî don't refactor broadly without reason.
2. If you touch anything in `rest/`, regenerate mocks (`make mock`) and run `make test`.
3. If you add a subcommand, add it under `cmd/internal/<name>/` and register it in `cmd/root.go`.
4. If you change onboarding/k8s behavior, update `docs/discovery-mode.md`.
5. Update `README.md` per `CONTRIBUTING.md` when changing user-visible interface (env vars, flags, ports).
6. Run `make test` before claiming work is done. CI is `make` + `make mock` + `gotestsum`.
